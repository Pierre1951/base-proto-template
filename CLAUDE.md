## 設計原則

以下はプロジェクト種別に関係なく厳守すべき原則です。

### ロジック層と UI 層の分離

検証したいロジックは純粋関数として書き、外部依存 (I/O、時間、乱数、グローバル状態) を直接参照しません。必要な場合は引数経由で注入します。

UI / プレゼンテーション層は検証結果をテキスト出力する最小限の層に留めます。本格的な UI は作らず、ドメインロジックの挙動観察に集中します。

### テスト駆動

ドメインの全ての公開関数に対応するテストを置きます。副作用を注入式にすることでテストの決定性を担保します。

### 型の厳密性

言語が型システムを持つ場合は最大強度で有効化します。動的型言語の場合は型チェッカー (mypy / pyright 等) を CI に組み込みます。

## 禁止事項

- グラフィックス・画像・音声アセットの導入
- 認証情報 (`.env`、キーファイル等) のコミット
- 依存管理ディレクトリ (`node_modules`、`venv`、`target` 等) のコミット
- ビルド成果物 (`dist`、`build` 等) のコミット

## 変更フロー

1 つの Issue に対して 1 つの PR を作成します。PR のコミットメッセージは Conventional Commits (`feat:`, `fix:`, `refactor:`, `test:`, `chore:`) に準拠します。

CI の `quality` チェックが通らない限り auto-merge は発火しません。CI が落ちた場合は同じ PR ブランチに修正コミットを追加します。

### PR 作成後の手順

PR を作成したら、以下コマンドで auto-merge を必ず有効化してください。

```bash
gh pr merge --auto --squash --delete-branch
```

CI が green なら自動的に main へマージされます。
